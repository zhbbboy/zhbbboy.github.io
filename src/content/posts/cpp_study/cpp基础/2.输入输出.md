## 输入输出

### 为什么 C++ 要重新做一套 I/O

C 语言的 `printf`/`scanf` 有两个根本问题。第一是**类型不安全**——格式字符串和实际参数的类型是分离的，`printf("%d", 3.14)` 编译能过但运行结果是垃圾值，编译器不会报错（现代编译器会 warning，但语言层面没有保障）。第二是**不可扩展**——你没法让 `printf` 直接输出一个自定义的 `Vector3` 类型，只能手动拆成三个 float 分别打印。

C++ 的 iostream 通过运算符重载解决了这两个问题：`<<` 和 `>>` 根据操作数类型自动选择正确的重载版本，既类型安全又可以为自定义类型扩展。

### 四个标准流对象
```cpp
#include <iostream>

// 这四个对象在程序启动时自动创建，不需要你手动构造
std::cout   // 标准输出，对应 C 的 stdout，默认连接到终端屏幕
std::cin    // 标准输入，对应 C 的 stdin，默认连接到键盘
std::cerr   // 标准错误，对应 C 的 stderr，无缓冲，立即输出
std::clog   // 标准日志，对应 C 的 stderr，有缓冲
```

`cout` 和 `clog` 是**有缓冲**的——数据先写入缓冲区，满了或遇到 `flush`/`endl` 才真正输出。`cerr` 是**无缓冲**的——每次写入都会立即刷新，保证错误信息不会因为程序崩溃而丢失。

面试考点："cout 和 cerr 的区别是什么？"——除了缓冲行为不同外，它们连接的文件描述符不同（cout 是 stdout，cerr 是 stderr），所以可以被 shell 分别重定向：`./game > output.log 2> error.log`。

### `<<` 和 `>>` 的本质

这两个不是什么神奇语法，就是**运算符重载**。
```cpp
std::cout << 42;
// 等价于调用：
// std::ostream::operator<<(int)
// 也就是 cout.operator<<(42)
```

标准库为 `ostream` 类预定义了一系列 `operator<<` 的重载版本：接受 `int`、`double`、`const char*`、`bool` 等。编译器根据你传入的实参类型自动匹配最合适的那个重载——这就是"类型安全"的来源。

#### 链式调用的原理

cpp

```cpp
std::cout << "x = " << 42 << std::endl;
```

这能链式写的原因是每个 `operator<<` 的返回值都是 `ostream&`（对自身的引用）：

cpp

```cpp
// 简化表示
ostream& operator<<(ostream& os, const char* str) {
    // 把 str 写入 os 的缓冲区
    return os;  // 返回引用，使得下一个 << 可以继续操作同一个流
}
```

所以链式调用展开后是：

cpp

```cpp
((std::cout << "x = ") << 42) << std::endl;
//  ↑ 返回 cout 的引用    ↑ 还是 cout 的引用
```

这个机制本身就是一道面试题——"为什么 `operator<<` 要返回引用？" 答：为了支持链式调用，并且避免拷贝流对象（流对象不可拷贝）。

### endl 和 '\n' 的区别

cpp

```cpp
std::cout << "hello" << std::endl;  // 输出换行 + 刷新缓冲区
std::cout << "hello" << '\n';       // 只输出换行，不刷新缓冲区
```

`std::endl` 等价于先输出 `'\n'` 再调用 `flush()`。在大量输出的场景下（比如游戏中的日志），频繁刷新缓冲区是有性能开销的。所以如果不需要保证立即输出，用 `'\n'` 就够了。

面试考点："endl 和 \n 有什么区别？性能上呢？"——endl 多一次 flush 操作，在循环中大量使用 endl 会明显变慢。

### cin 的输入机制与陷阱

#### 基本使用

cpp

```cpp
int a;
double b;
std::cin >> a >> b;  // 从输入流中提取一个 int 和一个 double
```

`>>` 默认以**空白字符**（空格、制表符、换行）为分隔符，自动跳过前导空白。

#### 陷阱一：cin >> 读字符串时遇到空格就停

cpp

```cpp
char name[100];
std::cin >> name;  // 输入 "John Smith"，name 只会得到 "John"
```

因为 `>>` 遇到空格就认为一个 token 读完了。要读一整行，需要用 `getline`：

cpp

```cpp
std::string name;
std::getline(std::cin, name);  // 读取一整行，包含空格
```

#### 陷阱二：cin >> 和 getline 混用

这是最经典的坑：

cpp

```cpp
int age;
std::string name;

std::cin >> age;              // 输入 "25\n"，>> 提取了 25，但 '\n' 留在缓冲区
std::getline(std::cin, name); // getline 读到残留的 '\n'，立即返回空字符串！
```

解决方案：在 `cin >>` 之后调用 `cin.ignore()` 丢弃残留的换行符：

cpp

```cpp
std::cin >> age;
std::cin.ignore(std::numeric_limits<std::streamsize>::max(), '\n');  // 清除到换行
std::getline(std::cin, name);  // 现在能正确读取了
```

或者简单写 `std::cin.ignore()` 也够用（默认忽略一个字符）。

#### 陷阱三：输入失败与流状态

cpp

```cpp
int x;
std::cin >> x;  // 用户输入了 "abc"
```

此时 `cin` 进入**失败状态**（failbit 被置位），后续所有 `>>` 操作都会静默失败。你必须手动清除错误状态：

cpp

```cpp
if (std::cin.fail()) {
    std::cin.clear();  // 清除错误标志
    std::cin.ignore(std::numeric_limits<std::streamsize>::max(), '\n');  // 丢弃坏数据
}
```

流对象有四个状态标志位：

```
goodbit  = 0    // 一切正常
eofbit   = 1    // 读到了文件末尾
failbit  = 2    // 逻辑错误（类型不匹配等）
badbit   = 4    // 严重的I/O错误（硬件故障等）
```

可以用 `cin.good()`、`cin.eof()`、`cin.fail()`、`cin.bad()` 来检查。流对象还重载了 `operator bool()`，所以可以直接当条件用：

cpp

```cpp
while (std::cin >> x) {  // 读取成功就继续，失败自动退出循环
    // 处理 x
}
```

### 格式控制

#### 通过流操纵器（manipulator）

cpp

```cpp
#include <iomanip>

std::cout << std::hex << 255 << std::endl;           // "ff"（十六进制）
std::cout << std::oct << 255 << std::endl;           // "377"（八进制）
std::cout << std::dec << 255 << std::endl;           // "255"（恢复十进制）

std::cout << std::fixed << std::setprecision(2) << 3.14159 << std::endl;  // "3.14"
std::cout << std::setw(10) << 42 << std::endl;       // "        42"（右对齐补空格）
std::cout << std::setfill('0') << std::setw(5) << 42 << std::endl;  // "00042"
std::cout << std::boolalpha << true << std::endl;     // "true"（而不是 "1"）
```

注意 `setw` 是**一次性的**——只对紧接着的下一个输出生效。而 `hex`、`fixed`、`setprecision` 是**持久性的**——设置后一直有效直到你改回来。这是一个常考的细节。

#### 操纵器的本质

`std::endl` 和 `std::hex` 这些其实都是函数（或函数对象），`operator<<` 有一个接受函数指针的重载版本：

cpp

```cpp
// 简化版，标准库中的实际定义
ostream& operator<<(ostream& os, ostream& (*manip)(ostream&)) {
    return manip(os);  // 把流对象传给这个函数
}

// endl 的实现大致是：
ostream& endl(ostream& os) {
    os.put('\n');
    os.flush();
    return os;
}
```

所以 `cout << endl` 实际上是把 `endl` 这个函数指针传给了 `operator<<`，后者调用它。这个机制面试偶尔会问。

### 为自定义类型重载 << 和 >>

这是 iostream 的核心优势——可扩展性：

cpp

```cpp
struct Vector3 {
    float x, y, z;
};

// 必须用友元或非成员函数，因为左操作数是 ostream 不是 Vector3
std::ostream& operator<<(std::ostream& os, const Vector3& v) {
    os << "(" << v.x << ", " << v.y << ", " << v.z << ")";
    return os;  // 别忘了返回引用
}

std::istream& operator>>(std::istream& is, Vector3& v) {
    is >> v.x >> v.y >> v.z;
    return is;
}

// 使用
Vector3 pos{1.0f, 2.0f, 3.0f};
std::cout << "Position: " << pos << std::endl;  // "Position: (1, 2, 3)"
```

面试考点："`operator<<` 为什么不能是成员函数？"——因为如果是 `Vector3` 的成员函数，那调用方式就变成了 `pos << cout`（左操作数是 `this`），语义完全反了。所以必须写成非成员函数或友元函数，让 `ostream&` 做第一个参数。

### 文件 I/O

cpp

```cpp
#include <fstream>

// 写文件
std::ofstream ofs("config.txt");  // 构造时自动打开
if (ofs.is_open()) {
    ofs << "resolution=1920x1080" << '\n';
    ofs << "fullscreen=true" << '\n';
}  // 析构时自动关闭（RAII）

// 读文件
std::ifstream ifs("config.txt");
std::string line;
while (std::getline(ifs, line)) {
    std::cout << line << '\n';
}
// 不需要手动 close()，析构函数会处理
```

文件流的打开模式：

```
ios::in       只读
ios::out      只写（默认截断文件）
ios::app      追加模式
ios::binary   二进制模式
ios::trunc    打开时截断（清空文件）
ios::ate      打开后定位到文件末尾
```

可以组合使用：`std::ofstream ofs("data.bin", ios::out | ios::binary);`

### 字符串流（stringstream）

把字符串当流来操作，非常适合做类型转换和字符串解析：

cpp

```cpp
#include <sstream>

// 数值 → 字符串
std::ostringstream oss;
oss << "Score: " << 9999 << " HP: " << 75.5f;
std::string result = oss.str();  // "Score: 9999 HP: 75.5"

// 字符串 → 数值
std::istringstream iss("100 200 300");
int a, b, c;
iss >> a >> b >> c;  // a=100, b=200, c=300
```

在 C++11 之后，简单的数值转换可以用 `std::to_string()` 和 `std::stoi()`/`std::stof()` 等，但 stringstream 在复杂格式解析中仍然不可替代。

### printf vs cout：性能与选择

这是游戏开发面试中一个实际的考量：

cpp

```cpp
// 默认情况下，cout 和 printf 的缓冲区是同步的（保证混用时输出顺序正确）
// 这种同步有性能代价

// 关闭同步可以显著加速 cout（竞赛编程常用技巧）
std::ios::sync_with_stdio(false);
std::cin.tie(nullptr);  // 解除 cin 和 cout 的绑定
```

`sync_with_stdio(false)` 之后，cout 的性能接近 printf。但代价是不能再混用 C 风格的 `printf`/`scanf` 和 C++ 的 `cout`/`cin`，否则输出顺序不可预测。

在实际游戏引擎中，UE 不直接用 `cout` 也不直接用 `printf`，而是用自己的 `UE_LOG` 宏，底层是格式化字符串 + 分级日志系统。了解 iostream 的原理是基础，但工程中要用引擎提供的日志接口。

### iostream 类层次结构

面试偶尔会问"cout 是什么类型的对象"或"iostream 的继承关系"：

```
          ios_base          ← 格式标志、状态标志等
             ↓
            ios             ← 流状态管理（good/fail/bad/eof）
           /   \
      istream   ostream    ← 输入/输出操作
         |    ×    |
         |  iostream |      ← 同时支持输入输出（菱形继承，用虚继承解决）
         |         |
      ifstream  ofstream    ← 文件流
      istringstream  ostringstream  ← 字符串流
```

`cin` 是 `istream` 类型的对象，`cout`/`cerr`/`clog` 是 `ostream` 类型的对象。`fstream` 和 `stringstream` 继承自 `iostream`（同时具备输入输出能力）。这里用到了**虚继承**来解决菱形继承问题——这个知识点会在讲继承与多态时详细展开。

### 面试要点小结

关于 C++ 输入输出，面试中真正会考的就这几个点：`operator<<` 为什么返回引用（链式调用）、为什么不能做成员函数（左操作数是 ostream）、`endl` vs `'\n'`（flush 的性能开销）、`cin >>` 和 `getline` 混用的坑（缓冲区残留 `'\n'`）、`sync_with_stdio` 的作用。这些是高频考点，其他的格式控制之类了解即可。
