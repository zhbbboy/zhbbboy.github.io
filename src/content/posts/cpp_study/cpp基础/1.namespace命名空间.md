## 1.命名空间(namespace)
### 1.1 命名空间要解决什么问题
C 语言中所有函数和全局变量都处于同一个全局作用域。当工程规模变大，多个库或多个开发者的代码合并时，名字冲突（name collision）不可避免。比如你写了一个 `Init()` 函数，引入的第三方库里也有一个 `Init()`，编译器就不知道你要调用哪个了。
C++ 用 namespace 来划分名字的"领地"——同名的标识符只要处于不同 namespace，就互不干扰。
	
### 1.2 基本语法
```cpp
namespace GameEngine {
    int version = 1;
    
    void Init() {
        // 引擎初始化
    }
    
    class Renderer {
        // ...
    };
}
```
这样 `Init`、`version`、`Renderer` 就都属于 `GameEngine` 这个命名空间，不会和全局的同名标识符冲突。
### 1.3 三种使用方式
这是最基础也是面试最常问的部分——"你知道几种使用命名空间的方式，各有什么优缺点？"

**方式一：作用域限定符 `::`（最推荐）**

```cpp
int main() {
    GameEngine::Init();
    std::cout << GameEngine::version << std::endl;
}
```

每次使用都带上完整的 namespace 前缀。优点是绝对不会产生歧义，代码可读性最好——一眼就知道这个符号来自哪里。缺点是写起来比较长。

**方式二：`using` 声明（引入单个成员）**

```cpp
using GameEngine::Init;  // 只把 Init 引入当前作用域

int main() {
    Init();  // OK, 就是 GameEngine::Init
}
```

精确引入需要的那一个名字，粒度最细。在函数体内部使用是安全的，在头文件中使用要谨慎。

**方式三：`using namespace` 指令（引入整个命名空间）**

```cpp
using namespace GameEngine;  // 把 GameEngine 下所有名字都引入

int main() {
    Init();     // GameEngine::Init
    version;    // GameEngine::version
}
```

最省事，但也最危险——你把整个命名空间的所有名字都"倒"进了当前作用域，名字冲突的风险又回来了。

**关键原则**：永远不要在头文件（.h/.hpp）的全局作用域写 `using namespace`。因为任何 `#include` 这个头文件的翻译单元都会被污染。这是面试中非常经典的考点。在 .cpp 的函数体内部用 `using namespace std` 问题不大，但在头文件的全局作用域写就是严重的工程错误。

### 1.4 命名空间可以不连续

同一个命名空间可以分散在多个文件中，编译器会自动合并：
```cpp
// file1.cpp
namespace GameEngine {
    void Init() { /*...*/ }
}

// file2.cpp
namespace GameEngine {
    void Shutdown() { /*...*/ }  // 和 file1 中的 Init 属于同一个命名空间
}
```

这就是为什么标准库的 `std` 可以分散在 `<iostream>`、`<vector>`、`<string>` 等几十个头文件中却仍然是同一个命名空间。

### 1.5 嵌套命名空间
```cpp
namespace Company {
    namespace Engine {
        namespace Rendering {
            class Shader { /*...*/ };
        }
    }
}

// 使用
Company::Engine::Rendering::Shader s;
```

C++17 提供了简化写法：

```cpp
namespace Company::Engine::Rendering {
    class Shader { /*...*/ };
}
```

效果完全一样，只是语法更简洁。UE 源码中虽然不用这个 C++17 语法（因为 UE 长期兼容 C++14），但嵌套命名空间的概念到处都是。

### 1.6 匿名命名空间

```cpp
namespace {
    int internalCounter = 0;  // 只在当前翻译单元可见
    void HelperFunc() { /*...*/ }
}
```

匿名命名空间里的内容具有**内部链接**（internal linkage），效果等同于给每个声明加上 `static`，但它是 C++ 推荐的做法（因为 `static` 用在全局作用域是 C 风格的，且不能作用于类型声明）。

### 提问
面试考点："`static` 全局函数和匿名命名空间有什么区别？"——功能上几乎一样，都限制在当前翻译单元内可见。但匿名命名空间可以包含类型定义（class、struct、enum），而 `static` 只能修饰变量和函数。C++ 标准更推荐用匿名命名空间替代文件作用域的 `static`。

#### 命名空间别名

当命名空间嵌套太深时，可以起别名：
```cpp
namespace Render = Company::Engine::Rendering;

Render::Shader s;  // 等价于 Company::Engine::Rendering::Shader s;
```

这在大型工程中很常用，UE 中你也会看到类似的简化手段。

#### ADL（Argument-Dependent Lookup）

这是一个偏进阶但面试偶尔会考的知识点，也叫 Koenig Lookup。
```cpp
namespace Math {
    struct Vector3 { float x, y, z; };
    
    Vector3 Add(const Vector3& a, const Vector3& b) {
        return {a.x + b.x, a.y + b.y, a.z + b.z};
    }
}

int main() {
    Math::Vector3 v1{1,0,0}, v2{0,1,0};
    auto v3 = Add(v1, v2);  // 居然不用写 Math::Add 也能编译通过！
}
```

编译器发现 `Add` 的参数类型 `Vector3` 来自 `Math` 命名空间，就会自动到 `Math` 中去查找 `Add`。这就是 ADL——编译器根据**参数的类型所属的命名空间**来扩展查找范围。

这也是为什么 `std::cout << "hello"` 能工作的原因之一——`operator<<` 是在 `std` 中定义的，而 `cout` 的类型属于 `std`，所以 ADL 自动找到了正确的重载。

#### inline 命名空间（C++11）

cpp

```cpp
namespace Engine {
    inline namespace v2 {
        void Init() { /* v2 实现 */ }
    }
    namespace v1 {
        void Init() { /* v1 实现 */ }
    }
}

Engine::Init();      // 调用的是 v2::Init（inline 版本）
Engine::v1::Init();  // 显式调用旧版本
```

`inline namespace` 的成员会被提升到父命名空间中直接可见。标准库用这个机制做 ABI 版本管理（比如 `std::__1` 就是 libc++ 的 inline namespace）。面试中很少深入考，但知道它的存在和用途即可。
