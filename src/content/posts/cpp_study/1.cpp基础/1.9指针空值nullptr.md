##  C++98中的指针空值

在良好的C/C++编程习惯中，在声明一个变量的同时最好给该变量一个合适的初始值，否则可能会出现不可预料的错误。比如未初始化的指针，如果一个指针没有合法的指向，我们基本都是按如下方式对其进行初始化：
```cpp
int* p1 = NULL;
int* p2 = 0;
```

NULL其实是一个宏，在传统的C头文件(stddef.h)中可以看到如下代码：
```cpp
/** Define NULL pointer value */
#ifndef NULL
#ifdef __cplusplus
#define NULL    0
#else  /* __cplusplus */
#define NULL    ((void *)0)
#endif  /* __cplusplus */
#endif  /* NULL */
```
 可以看到，NULL可能被定义为字面常量0，也可能被定义为无类型指针(void*)的常量。但是不论采取何种定义，在使用空值的指针时，都不可避免的会遇到一些麻烦，例如：
```cpp
#include <iostream>
using namespace std;
void Fun(int p)
{
	cout << "Fun(int)" << endl;
}
void Fun(int* p)
{
	cout << "Fun(int*)" << endl;
}
int main()
{
	Fun(0);           //打印结果为 Fun(int)
	Fun(NULL);        //打印结果为 Fun(int)
	Fun((int*)NULL);  //打印结果为 Fun(int*)
	return 0;
}
```
程序本意本意是想通过Fun(NULL)调用指针版本的Fun(int* p)函数，但是由于NULL被定义为0，Fun(NULL)最终调用的是Fun(int p)函数。

注：在C++98中字面常量0，既可以是一个整型数字，也可以是无类型的指针(void*)常量，但编译器默认情况下将其看成是一个整型常量，如果要将其按照指针方式来使用，必须对其进行强制转换。
### C++11中的指针空值
 对于C++98中的问题，C++11引入了关键字nullptr。

注意：
 1、在使用nullptr表示指针空值时，不需要包含头文件，因为nullptr是C++11作为关键字引入的。
 2、在C++11中，sizeof(nullptr)与sizeof((void*)0)所占的字节数相同。
 3、为了提高代码的健壮性，在后序表示指针空值时建议最好使用nullptr。





在现代 C++ 中，`nullptr` 的引入是为了彻底解决 C 语言时代遗留下来的“指针空值隐患”。它是 C++11 中最具实际安全意义的特性之一。

要理解它，我们得先看看它的前身 `NULL` 到底犯了什么错。

## 1. 历史痛点：C 语言时代的 `NULL` 到底是什么？

在传统的 C 和旧版 C++ 中，如果我们想让一个指针指向空，通常会写：

```C++
int* p = NULL;
```

然而，`NULL` 的本质并不是一个指针，它只是一个**宏定义**。如果你去翻阅底层的头文件（如 `<cstring>`），会发现它的定义通常是这样的：

```C
#define NULL 0
// 或者在某些 C 环境下是：
#define NULL ((void*)0)
```

但在 C++ 中，由于强类型检查的原因，`void*` 隐式转换成其他指针会报错，所以 **C++ 中的 `NULL` 实际上就是字面量整数 `0`**。

这就导致了一个极其致命的混乱：**“零”既可以代表整数0，又可以代表空指针。**

## 2. `NULL` 引发灾难的经典场景：函数重载

当 `NULL`（本质是 `0`）遇上**函数重载**时，编译器的类型分流就会完全失控：

```C++
#include <iostream>

void testFunc(int x) {
    std::cout << "调用了 int 版本" << std::endl;
}

void testFunc(int* ptr) {
    std::cout << "调用了 int* 指针版本" << std::endl;
}

int main() {
    testFunc(0);    // 毫无疑问，调用 int 版本
    
    // 我们的本意是传入空指针，期望调用 int* 版本
    testFunc(NULL); // ❌ 结果输出: "调用了 int 版本"
    
    return 0;
}
```

**原因**：因为 `NULL` 就是 `0`，它的类型是 `int`。当编译器看到 `testFunc(NULL)` 时，它会优先匹配参数类型完美吻合的 `int` 版本，而不是指针版本。这种二义性在大型项目中极易埋下难以排查的逻辑 Bug。

## 3. 救星登场：`nullptr` 的本质是什么？

为了从根本上划清“整数 0”与“空指针”的界限，C++11 专门引入了全新的关键字 **`nullptr`**。

- **它不是整数**：`nullptr` 的底层类型是 `std::nullptr_t`。
- **它是纯正的指针类别**：它只能被隐式转换为**任意类型的指针类型**（如 `int*`, `Mesh*`, `void*` 等），但**绝对不能**被隐式转换为整数（除了布尔值 `bool` 条件判断）。

如果用 `nullptr` 重新运行上面的重载测试：

```C++
testFunc(nullptr); // ✅ 精准调用: "调用了 int* 指针版本"
```

此时编译器能够一眼认出这是一个纯正的指针空值，从而做出正确的选择。

## 4. `nullptr` 的高频使用场景与核心规则

### 规则一：安全的条件判断

在条件判断中，`nullptr` 会自动转换为 `false`，这使得判空写法依然自然：

```C++
int* pData = getGeometryGeometry();

if (pData != nullptr) { // 显式判空
    // 执行安全操作
}

if (pData) { // 隐式判空，依然有效（底层安全转换）
    // ...
}
```

### 规则二：彻底杜绝类型乱套

```C++
int x = nullptr;  // ❌ 编译报错！无法将 nullptr_t 转换为 int
int* p = nullptr; // ✅ 编译通过
```

## 5. 串联总结：现代 C++ 指针置空规范

在现代 C++ 编程中，关于“空”的规范已经非常明确：

1. **绝对不要再使用 `NULL` 或 `0` 来初始化指针**。
    
2. 所有需要表达空指针的场景，**一律强制使用 `nullptr`**。
    
3. 无论是在处理底层图形网格拓扑（如寻找邻接面的半边 `HalfEdge*` 发现边界为空时），还是游戏引擎对象的生命周期管理中，使用 `nullptr` 都能让你的函数重载、模板推导和接口设计变得极其安全和清晰。
