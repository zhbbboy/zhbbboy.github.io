## 类的6个默认成员函数

如果一个类中什么成员都没有，我们简称其为空类。但是空类中真的什么都没有吗？其实不然，任何一个类，即使我们什么都不写，类中也会自动生成6个默认成员函数。

这 6 个默认成员函数是建立对象生命周期管理（创建、销毁、拷贝、赋值）的基石。
### 6 个默认成员函数全景图

我们先通过一张逻辑关系图，整体看一下这 6 个函数各自管辖的职责：

1. **初始化和清理**：
    
    - **构造函数**：负责对象的初始化。
        
    - **析构函数**：负责对象的清理（资源释放）。
        
2. **拷贝和赋值（核心）**：
    
    - **拷贝构造函数**：用一个已存在的对象去**初始化**另一个新对象。
        
    - **赋值运算符重载**：把一个已存在对象的值**赋值**给另一个已存在的对象。
        
3. **移动语义（C++11 引入，追求性能极致）**：
    
    - **移动构造函数**：通过“资源窃取”转移临时对象的所有权。它接收一个非 const 的右值引用（T&&）作为源对象，通过直接复制源对象的内部资源指针（或句柄），实现资源所有权的无缝转移，从而在运行时免除全新的物理内存分配（`new`/`malloc`）和数据复制（`memcpy`）开销。
        
    - **移动赋值运算符重载**：通过“资源窃取”给已存在对象赋值。移动赋值运算符重载是类成员运算符 `operator=` 的右值引用版本。它用于将一个即将销毁的右值对象拥有的资源所有权，转移给一个已经初始化完毕的现有左值对象。
        

## 构造函数
### 构造函数的概念

**构造函数**：名字与类名相同,创建类类型对象时由编译器自动调用，保证每个数据成员都有 一个合适的初始值，并且在对象的生命周期 内只调用一次。

例如，以下日期类中的成员函数Date就是一个构造函数。当你用该日期类创建一个对象时，编译器会自动调用该构造函数对新创建的变量进行初始化。
```cpp
class Date
{
public:
	Date(int year = 0, int month = 1, int day = 1)// 构造函数
	{
		_year = year;
		_month = month;
		_day = day;
	}
	void Print()
	{
		cout << _year << "年" << _month << "月" << _day << "日" << endl;
	}
private:
	int _year;
	int _month;
	int _day;
};

```

### 构造函数的特性

在 C++ 中，构造函数有着异于常人的语法规则：

1. **函数名必须与类名完全相同**。
2. **没有任何返回值类型**（连 `void` 都不能写）。
3. **在对象实例化时，由编译器自动调用**，且在对象的整个生命周期中**只能被调用一次**。
4. **可以发生函数重载**（一个类可以有多个不同参数的构造函数，用于提供不同的初始化方式）。
5. 如果程序员没有手动编写任何构造函数，编译器会自动赠送一个**默认构造函数**（无参）；**但只要你写了任意一个构造函数，编译器赠送的默认构造函数就会立刻失效**。
6. **无参的构造函数、全缺省的构造函数以及我们不写编译器自动生成的构造函数都称为默认构造函数，并且默认构造函数只能有一个**
 初学C++时，你可能认为只有当我们不写，编译器自动生成的构造函数才被称为默认构造函数。其实并不是这样的，以下3种都叫做默认构造函数：
 1、我们不写，编译器自动生成的构造函数。
 2、我们自己写的无参的构造函数。
 3、我们自己写的全缺省的构造函数。

说到这里，你可能会想：既然在我们不写的情况下，编译器会自动生成一个构造函数，那我们就没有必要自己写构造函数了。这种想法是不对的。

看看以下代码：
```cpp
#include <iostream>
using namespace std;
class Date
{
public:
	void Print()
	{
		cout << _year << "年" << _month << "月" << _day << "日" << endl;
	}
private:
	int _year;
	int _month;
	int _day;
};
int main()
{
	Date d1; // 编译器将调用自动生成的默认构造函数对d1进行初始化
	d1.Print();
	return 0;
}

```

 嗯…最终d1当中的年月日都是随机值。
 到这里你可能会产生疑问：d1对象调用了编译器自动生成的构造函数后，d1对象的_year/_month/_day依旧是随机值，那这编译器自动生成的构造函数还有什么意义？
**编译器自动生成的构造函数机制：**
 1、编译器自动生成的构造函数对内置类型不做处理。
 2、对于自定义类型，编译器会再去调用它们自己的默认构造函数。

**总结一下：虽然在我们不写的情况下，编译器会自动生成构造函数，但是编译器自动生成的构造函数可能达不到我们想要的效果，所以大多数情况下都需要我们自己写构造函数。**


### 2. 构造函数的分类与全面演示

我们以一个在 3D 图形几何处理或物理引擎中必不可少的“点（Point2D）”类为例，来看看各种形态的构造函数：

```C++
#include <iostream>

class Point2D {
private:
    double m_x;
    double m_y;

public:
    // 1. 无参构造函数（也叫默认构造函数）
    Point2D() {
        m_x = 0.0;
        m_y = 0.0;
        std::cout << "调用无参构造函数" << std::endl;
    }

    // 2. 带参构造函数（重载）
    Point2D(double x, double y) {
        m_x = x;
        m_y = y;
        std::cout << "调用带参构造函数" << std::endl;
    }

    // 3. 拷贝构造函数（用已有的对象初始化新对象）
    Point2D(const Point2D& other) {
        m_x = other.m_x;
        m_y = other.m_y;
        std::cout << "调用拷贝构造函数" << std::endl;
    }
};

int main() {
    // 各种构造函数的调用语法：
    
    Point2D p1;          // ✅ 正确：调用无参构造。注意：千万不要写成 Point2D p1(); 
                         // 否则编译器会误认为这是一个名为 p1、返回值是 Point2D 的函数声明！

    Point2D p2(10.0, 20.0); // ✅ 正确：调用带参构造（括号法）
    Point2D p3 = Point2D(5.0, 5.0); // ✅ 正确：显式调用
    
    Point2D p4(p2);      // ✅ 正确：调用拷贝构造
    
    return 0;
}
```

### 3. 核心进阶：初始化列表（Initialization List）

这是编写 C++ 构造函数时**最专业、最高效**的传统写法。

在上面的例子中，我们在构造函数体内部写 `m_x = x;`。这在底层实际上**并不是初始化，而是“赋值”**。这意味着变量先在出生时持有一个随机值（初始化），随后进入花括号内才被覆盖修改（赋值），平白无故多了一步开销。

现代 C++ 强烈推荐使用**初始化列表**，让变量在诞生的瞬间就拿到目标值：

```C++
class Point2D {
private:
    double m_x;
    double m_y;
public:
    // 语法：在冒号后面，直接对成员变量进行初始化
    Point2D(double x, double y) : m_x(x), m_y(y) {
        // 此时花括号内部通常什么都不用写
    }
};
```

#### 为什么必须使用初始化列表？（三种强制场景）

在绝大多数情况下，用初始化列表只是为了性能优化。但在以下三种特殊场景下，如果你不用初始化列表，**编译器会直接报错，程序根本无法编译通过**：

1. **类里包含 `const` 常量成员变量**：常量必须在诞生时立刻初始化，一旦进入花括号内部就无法对其“赋值”了。
2. **类里包含“引用”成员变量**：引用同样必须在定义时绑定实体，不能滞后。
3. **类里包含其他自定义类对象，且该对象“没有默认（无参）构造函数”**：

```C++
class NoDefault {
public:
    NoDefault(int a) {} // 只有带参构造，没有无参构造
};

class Test {
private:
    const int m_id;
    int& m_ref;
    NoDefault m_obj; // 自定义类对象
public:
    // ❌ 错误写法：在花括号内赋值，编译直接炸裂
    // Test(int id, int& r) { m_id = id; m_ref = r; m_obj = NoDefault(1); }

    // ✅ 唯一正确的写法：必须走初始化列表
    Test(int id, int& r, int val) : m_id(id), m_ref(r), m_obj(val) {
        // 安全起飞
    }
};
```

#### 🧠 隐藏的坑：初始化列表的执行顺序

请记住一条铁律：**初始化列表里成员变量的初始化顺序，是按照它们在类中“声明的顺序”决定的，而不是你在初始化列表里写的先后顺序！**

```C++
class Bad {
private:
    int m_a;
    int m_b;
public:
    // 看起来是先用 x 初始化 m_b，再用 m_b 初始化 m_a？
    Bad(int x) : m_b(x), m_a(m_b) {} 
};
```

- **结果**：因为 `m_a` 在类中先被声明，所以编译器会**先初始化 `m_a`**。此时 `m_b` 还没有被赋值，还是个随机值，于是 `m_a` 拿到了一个随机值。随后 `m_b` 才被正常初始化为 `x`。
    
- **最佳实践**：为了避免这种隐蔽的 Bug，**永远让初始化列表的编写顺序与类中变量的声明顺序保持绝对一致**。
### 4. 隐式类型转换与 `explicit` 关键字

如果你的构造函数**只有一个参数**（或者除了一个参数外，其他参数都有默认值），它会触发 C++ 的**隐式类型转换**机制。

```C++
class Distance {
private:
    double m_meters;
public:
    Distance(double m) : m_meters(m) {}
};

void printDistance(Distance d) { ... }

int main() {
    // 隐式转换：编译器自动把 double 类型的 5.5 包装塞进了 Distance 的构造函数里
    printDistance(5.5); 
    
    Distance d = 10.0; // 看起来像赋值，其实触发了隐式构造
}
```

这种隐式转换有时候很方便，但在大型工程（如图形学算法的矩阵、向量交互）中，它极易引发程序员意料之外的类型乱套和隐蔽 Bug。

如果不希望这种自动转换发生，可以在构造函数前加上 **`explicit`（显式的）** 关键字：
```C++
class Distance {
public:
    // 加上 explicit，拒绝一切偷偷摸摸的隐式转换
    explicit Distance(double m) : m_meters(m) {}
};

// 此时：
// printDistance(5.5); ❌ 编译报错！类型不匹配
// Distance d = 10.0;  ❌ 编译报错！
Distance d(10.0);      // ✅ 只有这种显式调用的语法才允许通过
```

### 5. 总结

- 构造函数是对象诞生的仪式。
- 编写构造函数时，**能写初始化列表就无脑写初始化列表**，既安全又高效。
- 对于单参数的构造函数，为了接口的严谨性，**建议默认加上 `explicit`**，除非你非常明确地需要隐式转换的便利性。

## 析构函数
### 析构函数的概念
**析构函数**：与构造函数功能相反，析构函数负责完成对象的销毁，对象在销毁时会自动调用析构函数，完成类的一些资源清理工作。

我们知道当一个类对象销毁时，其中的局部变量也会随着该对象的销毁而销毁，例如，我们用日期类创建了一个对象d1，当d1被销毁时，对象d1当中的局部变量_year/_month/_day也会被编译器销毁。

但是这并不意味着析构函数没有什么意义。像栈(Stack)这样的类对象，当该对象被销毁时，其中动态开辟的栈并不会随之被销毁，需要我们对其进行空间释放，这时析构函数的意义就体现了。
### 析构函数的特性

**一、析构函数的函数名是在类名前加上字符‘~’**
```cpp
class Date
{
public:
	Date()// 构造函数
	{}
	~Date()// 析构函数
	{}
private:
	int _year;
	int _month;
	int _day;
};
```
**二、没有任何返回值类型，也没有任何形参。**

**三、对象生命周期结束时，C++编译器会自动调用析构函数**  
 这就大大降低了C语言中栈空间忘记释放问题的发生，因为当栈对象生命周期结束时，C++编译器会自动调用析构函数对其栈空间进行释放。
 当对象离开作用域（栈对象）或被 `delete`（堆对象）时，编译器会自动在底层调用它，程序员通常不需要（也不应该）显式去调用它。
**四、一个类有且只有一个析构函数。若未显示定义系统会自动生成默认的析构函数**
   **编译器自动生成的析构函数机制：**  
 1、编译器自动生成的析构函数对内置类型不做处理。  
 2、对于自定义类型，编译器会再去调用它们自己的默认析构函数。
**五、先构造的后析构，后构造的先析构**  
 因为对象是定义在函数中的，函数调用会建立栈帧，栈帧中的对象构造和析构也要符合先进后出的原则。
 ![](../../_Pasted_Image/Pasted%20image%2020260707152210.png)
 **六、不能发生函数重载**
 因为没有参数，所以一个类**有且只能有一个**析构函数。

### 2. 浅拷贝 vs 深拷贝导致的“析构崩溃惨案”

在处理几何拓扑或动态数组时，我们经常需要手动分配内存。下面这个经典的例子能帮你彻底看清，为什么有时候程序会在析构函数里莫名其妙地崩溃（Double Free）。
#### 崩溃代码示范：

```C++
#include <iostream>

class TextureBuffer {
public:
    int* m_indices;
    int m_size;

    TextureBuffer(int size) : m_size(size) {
        m_indices = new int[size]; // 构造时开辟堆内存
    }

    // 手动编写的析构函数
    ~TextureBuffer() {
        if (m_indices) {
            delete[] m_indices; // 释放堆内存
            std::cout << "析构成功，内存已归还！" << std::endl;
        }
    }
};

void doSomething() {
    TextureBuffer t1(100);
    TextureBuffer t2 = t1; // ❌ 致命隐患：触发了编译器默认的【浅拷贝】
} // 离开作用域，t2 和 t1 开始排队销毁
```

#### 🧠 深度揭秘崩溃原因：

1. `TextureBuffer t2 = t1;` 执行后，由于是浅拷贝，`t2.m_indices` 和 `t1.m_indices` 内部存储了**一模一样的堆内存地址**。
2. 当函数 `doSomething` 结束时，局部变量开始析构。假设 `t2` 先析构，它顺利执行了 `delete[] m_indices;`，那块堆内存被成功释放。
3. 紧接着 `t1` 开始析构，它再次执行 `delete[] m_indices;`。然而，这块内存已经被 `t2` 释过了，此时对一块已经释放的“僵尸内存”进行二次解约，操作系统会勃然大怒，直接弹窗报错：**`双重释放 (Double Free Detacted)`，程序瞬间崩溃！**

**最佳实践**：如果类里面涉及底层资源分配，必须依照“五大函数法则”，手写**深拷贝**或者**移动语义**，断绝这种地址共享。

### 3. 析构函数的触发时机（对象什么时候死？）

不同内存区域的对象，其析构函数的触发时机截然不同。看懂下面这个例子，你就掌握了 C++ 对象的生命周期掌控权：

```C++
#include <iostream>
class Log {
    std::string m_msg;
public:
    Log(std::string msg) : m_msg(msg) {}
    ~Log() { std::cout << m_msg << " 被销毁" << std::endl; }
};

Log g_log("1. 全局对象"); // 全局变量

int main() {
    std::cout << "--- 进入 main ---" << std::endl;
    
    Log s_log("2. 局部栈对象");
    
    Log* p_log = new Log("3. 动态堆对象");
    
    {
        Log block_log("4. 局部块对象");
    } // 👈 此时 4 离开大括号作用域，立刻析构！
    
    std::cout << "准备 delete 堆对象..." << std::endl;
    delete p_log; // 👈 此时 3 被手动 delete，立刻析构！
    
    std::cout << "--- main 即将结束 ---" << std::endl;
    return 0;
} // 👈 main 结束，2 局部栈对象析构
// 👈 程序完全退出，1 全局对象析构
```

#### 控制台精准输出顺序：

```Plaintext
1 全局对象在 main 之前构造
--- 进入 main ---
4. 局部块对象 被销毁
准备 delete 堆对象...
5. 动态堆对象 被销毁
--- main 即将结束 ---
6. 局部栈对象 被销毁
7. 全局对象 被销毁
```

### 4. 析构函数的致命大坑：绝对不能在析构函数里抛出异常！

这是工程开发中极具破坏力的一个隐形炸弹。C++ 明确规定：**析构函数绝对不能将异常抛出到函数外部（建议无脑加 `noexcept`）。**

#### 为什么？（栈展开危机 Stack Unwinding）

当程序因为某处发生异常而崩溃并尝试寻找 `catch` 块时，C++ 运行时会自动去清理沿途的所有局部栈对象，这个清理过程会依次触发它们的**析构函数**。

- 如果在清理 A 对象的析构函数时，你的析构函数内部又向外抛出了一个新的异常。
- 此时，内存中同时存在**两个激活的异常**。C++ 标准对此束手无策，为了防止整个内存状态彻底失控，运行时会直接摆烂，强行调用 `std::terminate()` 杀死整个进程。

**黄金法则**：在析构函数里写代码时（比如关闭文件、释放内存），如果某些操作可能失败，必须在析构函数**内部**用 `try-catch` 默默消化掉，绝对不能让异常漏到外面去。


## 拷贝构造函数
### 拷贝构造函数的概念
拷贝构造函数：只有单个形参，该形参是对本类类型对象的引用（一般常用从const修饰），在用已存在的类类型对象创建新对象时由编译器自动调用。

```cpp
#include <iostream>
using namespace std;
class Date
{
public:
	Date(int year = 0, int month = 1, int day = 1)// 构造函数
	{
		_year = year;
		_month = month;
		_day = day;
	}
	Date(const Date& d)// 拷贝构造函数
	{
		_year = d._year;
		_month = d._month;
		_day = d._day;
	}
private:
	int _year;
	int _month;
	int _day;
};
int main()
{
	Date d1(2021, 5, 31);
	Date d2(d1); // 用已存在的对象d1创建对象d2

	return 0;
}

```
### 拷贝构造函数的特性

**一、拷贝构造函数是构造函数的一个重载形式**  
 因为拷贝构造函数的函数名也与类名相同。  
 
**二、拷贝构造函数的参数只有一个且必须使用引用传参，使用传值方式会引发无穷递归调用**
![](../../_Pasted_Image/Pasted%20image%2020260707155621.png)
要调用拷贝构造函数就需要先传参，若传参使用传值传参，那么在传参过程中又需要进行对象的拷贝构造，如此循环往复，最终引发无穷递归调用。
![](../../_Pasted_Image/Pasted%20image%2020260707155726.png)
**小贴士**：自定义类型的对象进行函数传参时，一般推荐使用引用传参。使用传值传参也可以，但每次传参时都会调用拷贝构造函数。

**三、若未显示定义拷贝构造函数，系统将生成默认的拷贝构造函数**
```cpp
#include <iostream>
using namespace std;
class Date
{
public:
	Date(int year = 0, int month = 1, int day = 1)// 构造函数
	{
		_year = year;
		_month = month;
		_day = day;
	}
	void Print()
	{
		cout << _year << "年" << _month << "月" << _day << "日" << endl;
	}
private:
	int _year;
	int _month;
	int _day;
};
int main()
{
	Date d1(2021, 5, 30);
	Date d2(d1); // 用已存在的对象d1创建对象d2
	d1.Print();
	d2.Print();
	return 0;
}

```
打印结果：
![](../../_Pasted_Image/Pasted%20image%2020260707160139.png)
代码中，我们自己并没有定义拷贝构造函数，但编译器自动生成的拷贝构造函数最终还是完成了对象的拷贝构造。  
**编译器自动生成的拷贝构造函数机制：**  
 1、编译器自动生成的拷贝构造函数对内置类型会完成浅拷贝（值拷贝）。  
 2、对于自定义类型，编译器会再去调用它们自己的默认拷贝构造函数。

**四、编译器自动生成的拷贝构造函数不能实现深拷贝**  
上面说到，编译器自动生成的拷贝构造函数会对内置类型完成浅拷贝。对于以下这句代码，浅拷贝实际上就是将d1的内容完完全全的复制了一份拷贝给d2，所以说浅拷贝也叫做值拷贝。
 ```cpp
 Date d2(d1);// 用已存在的对象d1创建对象d2
 ```
 但某些场景下浅拷贝并不能达到我们想要的效果。例如，栈(Stack)这样的类，编译器自动生成的拷贝构造函数就不能满足我们的需求了：
 ```cpp
 Stack s1; 
 Stack s2(s1);// 用已存在的对象s1创建对象s2
 ```
代码中，我们的本意是用已存在的对象s1创建对象s2，但编译器自动生成的拷贝构造函数，完成的是浅拷贝，拷贝出来的对象s2将不能满足我们的要求。  
举个例子，现有以下栈(Stack)类：
```cpp
class Stack
{
public:
	Stack(int capacity = 4)
	{
		_ps = (int*)malloc(sizeof(int)* capacity);
		_size = 0;
		_capacity = capacity;
	}
	void Print()
	{
		cout << _ps << endl;// 打印栈空间地址
	}
private:
	int* _ps;
	int _size;
	int _capacity;
};

```
 我们可以看到，类中没有自己定义拷贝构造函数，那么当我们用已存在的对象来创建另一个对象时，将调用编译器自动生成的拷贝构造函数。看看以下代码运行结果：
 ```cpp
 int main()
{
	Stack s1;
	s1.Print();// 打印s1栈空间的地址
	Stack s2(s1);// 用已存在的对象s1创建对象s2
	s2.Print();// 打印s2栈空间的地址
	return 0;
}
 ```
 结果打印s1栈和s2栈空间的地址相同，这就意味着，就算在创建完s2栈后，我们对s1栈做的任何操作都会直接影响到s2栈。
 ![](../../_Pasted_Image/Pasted%20image%2020260707161000.png)
这是我们想要的效果吗？显然不是，我们希望在创建时，s2栈和s1栈中的数据是相同的，但是在创建完s2栈后，我们对s1栈和s2栈之间的任何操作能够互不影响。
 
而且这种情况下，还会出现对同一块空间释放多次的问题。若我们自己定义的析构函数是正确的情况下，当程序运行结束，s2栈将被析构，此时那块栈空间被释放，然后s1栈也要被析构，再次对那一块空间进行释放。

 可以看到，这种情况下编译器自动生成的拷贝构造函数就不能满足我们的要求了。
总结一下：
 1、像Date这样的类，需要的就是浅拷贝，那么编译器自动生成的拷贝构造函数就够用了，我们不需要自己写。
 2、像Stack这样的类，浅拷贝会导致析构两次、程序崩溃等问题，需要我们自己写对应的拷贝构造函数。


在 C++ 中，拷贝构造函数是一个极其特殊的构造函数。它的核心任务就是：**用一个已经存在的老对象，去初始化一个刚刚诞生的一模一样的新对象。**
### 1. 拷贝构造函数的语法长什么样？

拷贝构造函数的参数有着雷打不动的标准格式：

```C++
class 类名 {
public:
    // 标准的拷贝构造函数格式
    类名(const 类名& obj) {
        // 复制逻辑
    }
};
```

#### 🧠 两个致命的面试拷问：

1. **为什么参数必须加引用 `&`？（不加引用会怎样？）**
    - **答案**：如果去掉引用写成 `类名(const 类名 obj)`，这就变成了**传值参数**。而在 C++ 中，传值参数在进入函数时，必须先将实参**拷贝**一份给形参。这就意味着，为了调用拷贝构造函数，系统需要先进行一次拷贝；而为了进行那次拷贝，系统又得去调用拷贝构造函数……这就陷入了**无限递归死循环**，直到栈内存溢出（Stack Overflow）程序崩溃。因此，编译器强制规定必须传引用。
        
2. **为什么参数通常要加 `const`？**
    - **答案**：加 `const` 有两个好处。第一，确保在函数内部我们只会读取老对象的数据，**保证不会意外修改老对象**。第二，**可以接收“临时对象”（右值）**。如果不加 `const`，诸如 `MyClass m2 = MyClass();` 这样的写法就会因为无法绑定临时对象而编译报错。

### 2. 拷贝构造函数什么时候会被触发？

在工业开发中，有且仅有以下**三种经典场景**会引发编译器自动调用拷贝构造函数：

#### 场景一：显式或隐式地用一个对象去初始化另一个对象

这是最直观的触发方式。
```C++
Geometry g1; 
Geometry g2(g1); // 显式调用（括号法）
Geometry g3 = g1; // 隐式调用（赋值法，注意：这依然是初始化，不是赋值运算符）
```

#### 场景二：当对象作为函数的【传值参数】时

当你把一个对象直接传给一个函数（而不是传指针或引用）：

```C++
void renderMesh(Geometry geo) { // geo 是形参
    // 执行渲染
}

int main() {
    Geometry myGeo;
    renderMesh(myGeo); // 实参传给形参时，暗中触发 myGeo 的拷贝构造函数！
}
```

#### 场景三：当函数返回一个【值对象】时

当函数内部创建了一个局部对象，并将其作为返回值传回：

```C++
Geometry createBox() {
    Geometry box;
    return box; // 返回时，会触发拷贝构造函数，把 box 克隆一份扔给调用者
}
```

(注：现代编译器非常智能，在场景三中通常会触发返回值优化 RVO / NRVO 来省去这次拷贝，但在语法理论上它确实是触发拷贝构造的节点。)

### 3. 编译器默认生成的拷贝构造函数：浅拷贝

如果你自己在类里**不写**拷贝构造函数，编译器会大方地送你一个默认的。但正如我们上一节所说，它送你的是**浅拷贝**（只抄藏宝图地址，共用一个箱子）。

我们来看一段真实的浅拷贝报错代码：

```C++
#include <iostream>

class FloatArray {
public:
    float* m_ptr;
    int m_size;

    FloatArray(int size) : m_size(size) {
        m_ptr = new float[size]; // 动态申请堆内存
    }

    // 程序员没有写拷贝构造函数，编译器默认为我们生成类似这样的代码：
    // FloatArray(const FloatArray& other) {
    //     m_ptr = other.m_ptr; // ❌ 仅仅复制了指针地址！
    //     m_size = other.m_size;
    // }

    ~FloatArray() {
        delete[] m_ptr; // 析构时释放堆内存
    }
};

int main() {
    FloatArray arr1(10);
    FloatArray arr2 = arr1; // 触发默认浅拷贝，两者的 m_ptr 指向同一块内存
    
    return 0; 
} // ⚠️ 退出 main 函数时，arr2 先析构把内存 delete 了；
  // 随后 arr1 析构再次 delete 同一块内存，触发 Double Free 异常，程序崩溃！
```

### 4. 完美避坑指南：手动编写【深拷贝】构造函数

面对上述类中包含指针、动态内存分配（堆资源）的情况，我们必须自己动手，丰衣足食，写出**深拷贝**版本的拷贝构造函数：

```C++
#include <iostream>
#include <cstring>

class SafeFloatArray {
public:
    float* m_ptr;
    int m_size;

    SafeFloatArray(int size) : m_size(size) {
        m_ptr = new float[size];
    }

    // 🔥 手动编写【深拷贝】构造函数
    SafeFloatArray(const SafeFloatArray& other) {
        m_size = other.m_size; // 复制普通变量
        
        if (other.m_ptr != nullptr) {
            // 第一步：向操作系统申请一块全新的、独立的内存仓库
            m_ptr = new float[m_size];
            
            // 第二步：把老仓库里的“物资（具体数据）”原封不动地搬运到新仓库
            std::memcpy(m_ptr, other.m_ptr, sizeof(float) * m_size);
        } else {
            m_ptr = nullptr;
        }
        std::cout << "深拷贝构造成功，新老对象资源已彻底隔离！" << std::endl;
    }

    ~SafeFloatArray() {
        delete[] m_ptr;
    }
};

int main() {
    SafeFloatArray arr1(10);
    SafeFloatArray arr2 = arr1; // ✅ 成功调用我们写的深拷贝，各自拥有独立的堆内存
    
    return 0; 
} // 各自调用析构函数释放各自的内存，安然无恙，完美退出！
```

### 5. 什么时候该用默认的？什么时候必须手写？

你不需要对每个类都手写拷贝构造函数，请牢记下面的分流判定：

1. **直接用默认的（不手写）**：如果你的类成员全部是基础类型（`int`, `double`, `bool`），或者成员本身就支持深拷贝（如 `std::string`, `std::vector`）。
2. **必须手写深拷贝**：如果你的类内部涉及到了手动的 `new`、`malloc`、底层的图形/文件句柄分配，为了防止 Double Free 崩溃，必须手写。
3. **彻底禁用拷贝**：有时候，某些独一无二的对象（例如游戏里的“玩家单例管理类”，或者几何内核里的“全局拓扑树”）是**不允许被复制**的，你可以在类里这样声明，直接把拷贝功能彻底咔嚓掉：
```C++
   class SystemManager {
   public:
   SystemManager(const SystemManager& other) = delete; // 显式禁用拷贝
  };
```

## 赋值运算符重载
### 运算符重载
C++为了增强代码的可读性引入了运算符重载，运算符重载是具有特殊函数名的函数，其目的就是**让自定义类型可以像内置类型一样可以直接使用运算符进行操作。**
```cpp
d1 == d2;// 可读性高（书写简单）
IsSame(d1, d2);// 可读性差（书写麻烦）
```
运算符重载函数也具有自己的返回值类型，函数名字以及参数列表。其返回值类型和参数列表与普通函数类似。  
运算符重载函数名为：关键字operator后面接需要重载的操作符符号。  

**函数原型：返回值 operator运算符(参数列表)**

注意：
 1.不能通过连接其他符号来创建新的操作符：比如operator@。
 2.重载操作符必须有一个类类型或枚举类型的操作数。
 3.用于内置类型的操作符，重载后其含义不能改变。
 4.作为类成员的重载函数时，函数有一个默认的形参this，限定为第一个形参。
 5.sizeof 、:: 、.* 、?: 、. 这5个运算符不能重载。

**这里以重载 == 运算符作为例子：**  
我们可以将该运算符重载函数作为类的一个成员函数，此时该函数的第一个形参默认为this指针。
```cpp
class Date
{
public:
	Date(int year = 0, int month = 1, int day = 1)
	{
		_year = year;
		_month = month;
		_day = day;
	}
	void Print()
	{
		cout << _year << "年" << _month << "月" << _day << "日" << endl;
	}
	bool operator==(const Date& d)// 运算符重载函数
	{
		return _year == d._year
			&&_month == d._month
			&&_day == d._day;
	}
private:
	int _year;
	int _month;
	int _day;
};

```
我们也可以将该运算符重载函数放在类外面，但此时外部无法访问类中的成员变量，这时我们可以将类中的成员变量设置为共有(public)，这样外部就可以访问该类的成员变量了（也可以用友元函数解决该问题）。并且在类外没有this指针，所以此时函数的形参我们必须显示的设置两个。

```cpp
class Date
{
public:
	Date(int year = 0, int month = 1, int day = 1)
	{
		_year = year;
		_month = month;
		_day = day;
	}
	void Print()
	{
		cout << _year << "年" << _month << "月" << _day << "日" << endl;
	}
	int _year;
	int _month;
	int _day;
};
bool operator==(const Date& d1, const Date& d2)// 运算符重载函数
{
	return d1._year == d2._year
		&&d1._month == d2._month
		&&d1._day == d2._day;
}

```
### 赋值运算符重载
**这里以重载 = 运算符作为例子：**
```cpp
class Date
{
public:
	Date(int year = 0, int month = 1, int day = 1)// 构造函数
	{
		_year = year;
		_month = month;
		_day = day;
	}
	Date& operator=(const Date& d)// 赋值运算符重载函数
	{
		if (this != &d)
		{
			_year = d._year;
			_month = d._month;
			_day = d._day;
		}
		return *this;
	}
	void Print()// 打印函数
	{
		cout << _year << "年" << _month << "月" << _day << "日" << endl;
	}
private:
	int _year;
	int _month;
	int _day;
};

```
重载赋值运算符需要注意以下几点：
一、**参数类型设置为引用，并用const进行修饰**
 赋值运算符重载函数的第一个形参默认是this指针，第二个形参是我们赋值运算符的右操作数。
 由于是自定义类型传参，我们若是使用传值传参，会额外调用一次拷贝构造函数，所以函数的第二个参数最好使用引用传参（第一个参数是默认的this指针，我们管不了）。
 其次，第二个参数，即赋值运算符的右操作数，我们在函数体内不会对其进行修改，所以最好加上const进行修饰。
 
二、**函数的返回值使用引用返回**
 实际上，我们若是只以d2 = d1这种方式使用赋值运算符，赋值运算符重载函数就没必要有返回值，因为在函数体内已经通过this指针对d2进行了修改。但是为了支持连续赋值，即d3 = d2 = d1，我们就需要为函数设置一个返回值了，而且很明显，返回值应该是赋值运算符的左操作数，即this指针指向的对象。
 和使用引用传参的道理一样，为了避免不必要的拷贝，我们最好还是使用引用返回，因为此时出了函数作用域this指针指向的对象并没有被销毁，所以可以使用引用返回。
 
三、**赋值前检查是否是给自己赋值**
 若是出现d1 = d1，我们不必进行赋值操作，因为自己赋值给自己是没有必要进行的。所以在进行赋值操作前可以先判断是否是给自己赋值，避免不必要的赋值操作。
 
**四、引用返回的是*this**
 赋值操作进行完毕时，我们应该返回赋值运算符的左操作数，而在函数体内我们只能通过this指针访问到左操作数，所以要返回左操作数就只能返回*this。
 
**五、一个类如果没有显示定义赋值运算符重载，编译器也会自动生成一个，完成对象按字节序的值拷贝**
 没错，赋值运算符重载编译器也可以自动生成，并且也是支持连续赋值的。但是编译器自动生成的赋值运算符重载完成的是对象按字节序的值拷贝，例如d2 = d1，编译器会将d1所占内存空间的值完完全全地拷贝到d2的内存空间中去，类似于memcpy。
 对于日期类，编译器自动生成的赋值运算符重载函数就可以满足我们的需求，我们可以不用自己写。但是这也不意味着所有的类都不用我们自己写赋值运算符重载函数，当遇到一些特殊的类，我们还是得自己动手写赋值运算符函数的。
**注意区别以下代码所调用的函数：**
```cpp
	Date d1(2021, 6, 1);
	Date d2(d1);
	Date d3 = d1;

```
这里一个三句代码，我们现在都知道第二句代码调用的是拷贝构造函数，那么第三句代码呢？调用的是哪一个函数？是赋值运算符重载函数吗？
 其实第三句代码调用的也是拷贝构造函数，注意区分拷贝构造函数和赋值运算符重**载函数的使用场景(重点)：**
 **拷贝构造函数**：用一个已经存在的对象去构造初始化另一个即将创建的对象。
 **赋值运算符重载函数**：在两个对象都已经存在的情况下，将一个对象赋值给另一个对象。


### 赋值运算符的核心任务是：**把一个已经存在的老对象的值，复制给另一个“同样已经存在”的老对象。**

### 1. 语法长什么样？

赋值运算符重载的本质是**运算符重载**。它的编写格式极其讲究，不仅参数要加 `const&`，返回值也死死限定为**当前类的引用**。

标准的编写范式如下：
```C++
class MyClass {
public:
    // 标准赋值运算符重载格式
    MyClass& operator=(const MyClass& other) {
        // 1. 防范自我赋值
        if (this == &other) {
            return *this; 
        }

        // 2. 释放自己的老资源（如果有堆内存）
        
        // 3. 复制新物资（深拷贝）

        // 4. 返回 *this
        return *this;
    }
};
```

### 2. 核心细节逐层剥离（面试高频）

为什么一个简单的赋值要写得这么繁琐？

#### 疑问一：为什么返回值必须是 `MyClass&`（引用）？

- **答案**：为了**支持连续链式赋值**（形如 `a = b = c;`）。
- 在 C++ 中，内置类型是支持连续赋值的。执行 `a = b = c;` 时，编译器会先执行 `b = c`，这一步会返回 `b` 的引用，接着代码演变成 `a = b`。
- 如果你的返回值是 `void`，那写下 `a = b = c;` 时，`b = c` 完事后返回个寂寞，外层的 `a =` 找不到操作数，**编译器就会直接报错**。
    

#### 疑问二：为什么一进来就要判断 `if (this == &other)`？

- **答案**：这是为了**防止自我赋值（Self-assignment）引发的自杀惨案**。
- 程序员有时候会无意中写出 `g1 = g1;` 或者通过指针间接写出 `*ptr1 = *ptr2;`（其实它们指向同一个对象）。
- 如果不加这个判断，按照后面的逻辑，你会**先把自己的老资源给 delete 清理掉**。由于 `other` 就是你自己，这就意味着 `other` 的资源也同步被你销毁了。等下一步你去搬运 `other` 的物资时，面对的只有一堆已经死掉的僵尸垃圾，程序会瞬间崩溃。

### 3. 经典区分：拷贝构造 vs 赋值重载

很多初学者容易把它们俩搞混，因为它们的语法长得太像了。

 **辨析法则**：**看符号左边的对象，在这一行执行之前，到底“出生”了没有。**

```C++
Geometry g1;          // 构造函数

Geometry g2 = g1;     // 👈 这是【拷贝构造函数】！
 // 辨析：g2 此时才刚刚出生，是用 g1 来初始化它。

Geometry g3;          // 构造函数
g3 = g1;              // 👈 这才是【赋值运算符重载】！
         // 辨析：g3 早就出生了，这是两个成年的老对象之间在复制物资。
```

### 4. 手动编写【深拷贝】赋值重载

如果你的类里包含了**指针或堆区动态开辟的内存**，默认的赋值运算符依然是危险的浅拷贝。我们必须亲自动手，严格按照四个步骤，写出绝对安全的深拷贝赋值：

```C++
#include <iostream>
#include <cstring>

class SafeArray {
public:
    int* m_ptr;
    int m_size;

    SafeArray(int size) : m_size(size) {
        m_ptr = new int[size];
    }

    // 析构函数
    ~SafeArray() {
        delete[] m_ptr;
    }

    // 🔥 完美手写【深拷贝】赋值运算符重载
    SafeArray& operator=(const SafeArray& other) {
        // 第一步：如果是自己给自己赋值，直接打道回府，绝不折腾
        if (this == &other) {
            return *this;
        }

        // 第二步：由于自己是已经存在的老对象，肚子里可能装有老物资，
        // 必须先把自己的老内存 delete 掉，否则直接换新箱子会导致老内存泄漏！
        delete[] m_ptr;

        // 第三步：开始正规的深拷贝流程
        m_size = other.m_size;
        if (other.m_ptr != nullptr) {
            // 重新开辟一块干净的、完全属于自己的新内存仓库
            m_ptr = new int[m_size];
            // 把对方仓库里的实际物资克隆过来
            std::memcpy(m_ptr, other.m_ptr, sizeof(int) * m_size);
        } else {
            m_ptr = nullptr;
        }

        std::cout << "赋值运算符深拷贝成功！" << std::endl;

        // 第四步：雷打不动，返回对象本身
        return *this;
    }
};

int main() {
    SafeArray arr1(10);
    SafeArray arr2(20);
    
    arr2 = arr1; // ✅ 安全赋值！arr2 成功释放了旧的 20 大小的内存，并克隆了 arr1 的数据
    
    return 0;
}
```

### 5. 现代 C++ 进阶：拷贝交换技术

在现代 C++ 开发中，为了追求极端的**异常安全性（Exception Safety）**，大佬们往往不屑于写上面那种传统的四步法，而是采用一种近乎艺术的写法——**Copy-and-Swap 炫技流**。

如果按照传统写法，万一第三步 `m_ptr = new int[m_size];` 因为系统内存不足**抛出了异常**，此时你的老内存已经在第二步被 `delete` 了，整个对象直接变成了一个不合法的残废，状态彻底失控。

#### Copy-and-Swap 的写法：

```C++
#include <utility> // 包含 std::swap

class SafeArray {
public:
    int* m_ptr;
    int m_size;

    // 假设已经写好了完美的拷贝构造函数
    SafeArray(const SafeArray& other) { ... }

    // 现代赋值运算符重载：注意看，形参直接写传值「SafeArray other」，不写引用！
    SafeArray& operator=(SafeArray other) { // 👈 绝妙之处：进函数时自动触发【拷贝构造】，天然生成一份安全的临时副本
        
        // 悄悄把这个临时副本里的一切物资，和当前自己（this）的所有资产进行对调！
        std::swap(m_ptr, other.m_ptr);
        std::swap(m_size, other.m_size);

        return *this;
    } // 👈 函数结束，other 这个临时工带着你原本的老垃圾资产，高高兴兴地走出花括号，触发析构函数被自动物理销毁！
};
```

#### 这种写法强在哪里？

1. **天然防范自我赋值**：如果是自我赋值，进函数时拷贝一份，自己跟副本交换，完全不伤及本尊。
2. **绝对异常安全**：由于新内存是在传参阶段的拷贝构造里申请的。如果内存不足抛出异常，压根连函数大门都进不来，当前对象的原有数据**完好无损**。
3. **极简优雅**：一行 `delete` 都不用写，全靠“临时工”替你背黑锅销毁资源。


## const成员
### const修饰类的成员函数
我们将const修饰的类成员函数称之为const成员函数，const修饰类成员函数，实际修饰的是类成员函数隐含的this指针，表明在该成员函数中不能对this指针指向的对象进行修改。

例如，我们可以对类成员函数中的打印函数进行const修饰，避免在函数体内不小心修改了对象：
```cpp
	void Print()const// cosnt修饰的打印函数
	{
		cout << _year << "年" << _month << "月" << _day << "日" << endl;
	}

```

思考下面几个问题（经典面试题）：
1.const对象可以调用非const成员函数吗？
2.非const对象可以调用const成员函数吗？
3.const成员函数内可以调用其他的非const成员函数吗？
4.非cosnt成员函数内可以调用其他的cosnt成员函数吗？

答案是：不可以、可以、不可以、可以
解释如下：
1.非const成员函数，即成员函数的this指针没有被const所修饰，我们传入一个被const修饰的对象，用没有被const修饰的this指针进行接收，属于权限的放大，函数调用失败。

2.const成员函数，即成员函数的this指针被const所修饰，我们传入一个没有被const修饰的对象，用被const修饰的this指针进行接收，属于权限的缩小，函数调用成功。

3.在一个被const所修饰的成员函数中调用其他没有被const所修饰的成员函数，也就是将一个被const修饰的this指针的值赋值给一个没有被const修饰的this指针，属于权限的放大，函数调用失败。

4.在一个没有被const所修饰的成员函数中调用其他被const所修饰的成员函数，也就是将一个没有被const修饰的this指针的值赋值给一个被const修饰的this指针，属于权限的缩小，函数调用成功。

### 取地址及const取地址操作符重载
取地址操作符重载和const取地址操作符重载，这两个默认成员函数一般不用自己重新定义，使用编译器自动生成的就行了：

class Date
{

在 C++ 中，**`const` 成员**是指在类中被 `const` 关键字修饰的成员。类中的 `const` 成员可以分为两大类：**`const` 成员变量** 和 **`const` 成员函数**。它们分别从“数据”和“行为”两个维度，为类拉起了安全的只读防护网。
### 一、 `const` 成员变量（只读属性）

`const` 成员变量的含义非常直接：**一旦某个对象的该属性被初始化，其值在整个对象的生命周期内绝对不允许被修改。**
#### 1. 唯一合法的初始化通道

普通的 `const` 变量可以在定义时直接赋值，但类的 `const` 成员变量**必须在对象诞生时进行初始化**。

正如我们前面在构造函数那一节所聊到的，`const` 成员变量**强制要求必须使用“初始化列表”进行初始化**，绝对不能在构造函数的花括号 `{}` 内部进行赋值。

```C++
class Circle {
public:
    const double pi; // const 成员变量
    double radius;

    // ❌ 错误写法：进入花括号内已经属于“赋值”阶段，常量拒绝被赋值
    // Circle(double r) { pi = 3.14159; radius = r; }

    // ✅ 唯一正确的写法：走初始化列表
    Circle(double r) : pi(3.14159), radius(r) {}
};
```

### 2. 内存特性

每个独立的类实例（对象）都有属于自己的 `const` 成员变量，它们存放在具体对象的物理内存中。这意味着，不同对象的 `const` 成员变量可以持有不同的值：

```C++
class Device {
public:
    const int id;
    Device(int deviceId) : id(deviceId) {} // 每个设备的 id 在诞生后都无法更改，但各不相同
};
```

## 二、 `const` 成员函数（只读行为）

`const` 成员函数是现代 C++ 接口设计（API）中极其重要的一环。它的核心承诺是：**该函数绝对不会修改当前对象的任何成员变量。**

### 1. 语法长什么样？

必须把 `const` 关键字挂在**函数的形参列表括号后面，函数体大括号前面**：
```C++
class Player {
private:
    int m_hp = 100;
public:
    // 🔥 const 成员函数：纯读取，不修改
    int getHp() const {
        // m_hp = 200; ❌ 报错！不允许修改成员变量
        return m_hp;
    }

    // 普通成员函数：允许修改
    void takeDamage(int damage) {
        m_hp -= damage; 
    }
};
```

### 2. `this` 指针的升级

我们在 `this` 指针那一节扒过它的本质。当你在成员函数后面加了 `const` 之后，编译器在底层转译该函数时，会**把隐式传入的 `this` 指针类型从 `Player* const` 升级为 `const Player* const`**。

因为 `this` 指针变成了指向常量的指针，任何通过 `this->m_hp = xxx` 的写入操作在编译阶段就会被直接卡死，从而在底层确保了对象的绝对安全。

## 三、 `const` 对象的权限调用规则

当 `const` 遇上类对象实例时，C++ 拉起了一条非常严苛的权限分流红线：

```C++
int main() {
    Player p1;          // 普通对象（非常量对象）
    const Player p2;    // const 对象（常量对象，属性只读）

    // 1. 普通对象 p1：拥有最高权限
    p1.getHp();         // ✅ 允许调用 const 成员函数（权限缩小，安全）
    p1.takeDamage(10);  // ✅ 允许调用普通成员函数

    // 2. const 对象 p2：权限受到死死锁定
    p2.getHp();         // ✅ 允许调用 const 成员函数
    // p2.takeDamage(10);  ❌ 编译直接报错！
}
```

### 核心权限规则表：
 - **普通对象** $\rightarrow$ 既可以调用普通成员函数，也可以调用 `const` 成员函数。
 - **`const` 对象** $\rightarrow$ **只能**调用 `const` 成员函数，绝对不允许调用任何普通成员函数！
 
**开发痛点**：如果你在写类的时候，有些“明明只是读一下数据”的函数（如 `getX()`, `printLog()`）**忘记加 `const` 修饰**。那么未来当其他开发者写出 `const MyClass& obj`（常量引用传参）去遍历或读取你的对象时，他们将**无法调用你这些忘记加 `const` 的只读函数**，直接引发编译大面积报错。

## 四、 两个高级特例：重载与突破

### 1. 依靠 `const` 也能构成函数重载！

由于 `const` 成员函数改变了隐式参数 `this` 指针的类型，因此**同名、同参数的函数，后面加不加 `const` 可以完美构成函数重载**。

```C++
class Array {
private:
    int data[10];
public:
    // 版本 A：给普通对象用，返回普通引用，允许外部通过下标修改内容
    int& operator[](int index) { return data[index]; }

    // 版本 B：给 const 对象用，返回常量引用，外部只能读不能写
    const int& operator[](int index) const { return data[index]; }
};
```

编译器会根据调用对象本身是不是 `const`，精准分流到对应的版本中。

### 2. 突破限制：`mutable` 关键字

在极少数业务场景下，我们希望在 `const` 成员函数内部，去修改某个与业务逻辑无关的、属于“幕后工作”的成员变量（例如：统计该只读函数被调用了多少次的计数器 `m_callCount`，或者是缓存数据的锁）。

你可以使用 **`mutable`（可变的）** 关键字来为该变量特赦：
```C++
class DataKernel {
private:
    int m_data = 0;
    mutable int m_readCount = 0; // 特赦变量

public:
    int getData() const {
        m_readCount++; // ✅ 允许！因为变量被标记为了 mutable，即使在 const 函数里也能修改
        return m_data;
    }
};
```

## 五、 总结

在 C++ 面向对象中，`const` 成员是高质量代码的试金石：

- **`const` 成员变量**：通过初始化列表一次锁定，终身只读。
- **`const` 成员函数**：升级底层 `this` 指针，对 `const` 对象张开双臂。
- **开发信条**：**只要一个成员函数内部不需要修改成员变量，请毫不犹豫地在后面加上 `const`**。这不仅能保护数据，更能让你的类接口无缝契合各种常量引用传参的高性能场景。
